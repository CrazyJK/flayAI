## 0. 한 줄 요약

> 사내에 흩어진 **메일 · 게시물 · 결재문서**를, 권한을 지키면서 **자연어 한 문장으로 통합 검색**하는 사내 전용 RAG 시스템.
> 사내망·온프레미스 전용으로, 데이터를 외부로 내보내지 않는다.

### 용어 빠른 정의 (AI/ML 입문자용)

- **RAG (Retrieval-Augmented Generation)**: "먼저 검색해서 찾은 문서를 LLM 에게 보여주고 답하게 한다." LLM 단독이 아니라 *DB 검색 + LLM 종합*.
- **임베딩 (embedding)**: 문장을 의미를 담은 숫자 벡터(예: 1024차원)로 바꾼 것. 의미가 비슷하면 벡터도 가깝다 → "예산 회의" 와 "비용 검토" 를 키워드 없이도 연결.
- **하이브리드 검색**: 키워드 검색(BM25)과 의미 검색(벡터)을 **둘 다** 돌려 합치는 것. 정확한 이름·번호는 키워드가, 자연어 의도는 벡터가 잡는다.
- **리랭커 (reranker)**: 1차로 넓게 찾은 후보 수십 개를, 질의와 각 문서를 **함께** 정밀 비교해 다시 순위 매기는 모델(cross-encoder). 검색 정확도를 크게 올리는 RAG 표준 단계.
- **청킹 (chunking)**: 긴 문서를 검색 단위(문단/섹션)로 쪼개는 것. 결재문서·긴 메일은 통째로 임베딩하면 정확도가 떨어져 쪼갠다.

## 1. 배경 & 문제

- 사내 업무 정보가 **시스템별로 단절**되어 있다: 메일(메일서버), 공지·게시물(그룹웨어/게시판), 결재문서(전자결재 시스템).
- 찾을 때마다 시스템을 옮겨 다니며 키워드로 검색 → 정확한 제목/작성자/기간을 기억해야 하고, 자연어 의도("작년 4분기 예산 관련 결재 건")로는 못 찾는다.
- 결과적으로 **이미 사내에 있는 정보를 다시 묻거나, 못 찾아서 중복 작업**이 발생한다.
- 사내 데이터는 **외부 SaaS LLM 에 올릴 수 없다**(보안·컴플라이언스) → **온프레미스 로컬 LLM** 필수.

## 2. 목표 & 범위

### 2.1 목표

1. 메일·게시물·결재문서를 **한 검색창**에서 자연어로 검색.
2. 결과에 **출처(원문 링크·작성자·일자·시스템)**를 항상 표기, 가능하면 **요약 답변 + 인용**.
3. **권한을 100% 준수** — 사용자가 접근 권한 없는 문서는 검색·답변·요약 어디에도 노출 금지.
4. **멀티유저 동시 접속**에서 만족스러운 응답 속도(목표 SLA 는 §11).
5. 사내망 전용, 외부 인터넷 노출 금지(전송·저장 암호화, 감사 로그).

### 2.2 범위 (In Scope)

- 데이터 소스: **메일 · 게시물 · 결재문서** (1차). 첨부파일(pdf/docx/xlsx/hwp/이미지) 텍스트 추출 포함.
- 검색 모드: ① 검색(문서 카드+스니펫) ② RAG 답변(요약+인용) — 둘 다 또는 단계 도입(§4.2).
- 권한 기반 필터링, SSO 로그인, 감사 로그.

### 2.3 비범위 (Out of Scope, 1차 제외 — 결정 필요)

- 실시간 채팅/그룹웨어 대체, 문서 작성·결재 기능(검색 전용).
- 외부(인터넷) 정보 검색.
- `[TBD]` 음성 검색, 모바일 전용 앱, 다국어 UI 등.

## 3. 사용자 시나리오 (예시)

| 질의 예 | 기대 동작 |
|---------|-----------|
| "지난 분기 마케팅 예산 결재 문서 찾아줘" | 결재문서, 기간(분기)·부서 필터 + 의미검색 → 카드 + 요약 |
| "김부장이 보낸 보안 점검 관련 메일" | 메일, 발신자=김부장 + 주제 의미검색 |
| "재택근무 정책 최신 공지" | 게시물, 의미검색 + 최신순 |
| "이 프로젝트 관련 메일·결재·게시물 다 모아줘" | 소스 통합 검색, 프로젝트명 의미검색 |
| "작년 12월 출장비 정산 어떻게 했더라" | 결재+메일, 기간 필터 + RAG 요약(인용) |

> **권한 전제**: 위 모든 질의는 *질의한 사용자가 볼 수 있는 문서 안에서만* 동작한다.

## 4. 핵심 설계 원칙

### 4.1 검색 코어

- **하이브리드 검색**: 의미 검색(벡터)과 키워드 검색(BM25, 한국어 형태소)을 함께 수행하고 **RRF(Reciprocal Rank Fusion)** 로 결합. 정확한 이름·번호는 키워드가, 자연어 의도는 벡터가 보완한다.
- **리랭커 추가**: RRF 상위 후보를 cross-encoder 로 재정렬해 정확도를 끌어올린다.
- **권한 우선(permission-first)**: 검색의 모든 단계에서 사용자가 볼 수 있는 문서만 후보로 둔다(§9).
- **증분·멱등 인덱싱**: 변경분만 반영하고 단계별 재실행이 안전하도록 설계 → 대량 초기 인덱싱과 운영 동기화를 안정적으로.
- **로컬 LLM**: 질의·문서가 외부로 나가지 않도록 온프레미스 추론.

> 본 시스템에서 새로 풀어야 하는 핵심 과제는 **① 다양한 사내 소스 연동 ② 권한 정확성 ③ 멀티유저 성능** 세 가지다. 검색 알고리즘 자체보다 이 셋이 난이도를 좌우한다.

### 4.2 검색 모드 (2가지 — 단계 도입 가능)

- **모드 A: 검색(Retrieval-only)** — 문서 카드 + 강조 스니펫 + 출처. LLM 답변 없음(또는 1줄 요약). 빠르고 안전(환각 없음).
- **모드 B: RAG 답변(Generation)** — 상위 문서를 LLM 에 넣어 **요약 답변 + 인용 출처**. "요약해줘/정리해줘" 류에 유용하나 **환각·권한 누출 위험**이 있어 인용 강제·권한 필터 후처리가 필수.
- **결정 필요**: 1차는 모드 A 로 안전하게 출시 → 검증 후 모드 B 추가(권장). 또는 처음부터 B 포함.

## 5. 차별화 포인트 (특장점)

> 2025~2026년 기준 **하이브리드 검색 · 리랭킹 · 권한 인식**은 이미 업계의 *기본기(table stakes)* 에 가깝다. 차별성은 그 위에 더하는 아래 설계에서 나온다. 업계가 공통으로 지적하는 RAG 실패 원인(근거가 약할 때의 환각 / stale·삭제 누락 / 시간 정보 오류 / 권한 누출 / 인용 부재)을 최대한 줄이는 것을 목표로 한다.

### 5.1 한눈에 — 기본기 vs 본 시스템

| 이미 업계 표준 (기본기) | 본 시스템이 더하는 차별화 |
|--------------------------|----------------------------|
| 하이브리드 검색 + 리랭커 | + **결정론적 메타·시간 추출 가드레일** (§5.2-②) |
| 권한 인식 검색 | + **삭제·권한회수 전파** + 누출 테스트 상시 (§5.2-⑤) |
| 출처 인용 | + **근거 부족 시 답변 보류(abstain)** (§5.2-③) |
| 단일 소스별 검색 | + **크로스소스 엔티티·스레드 통합** (§5.2-①) |
| 첨부 OCR 텍스트 색인 | + **VLM 의미 캡션**(도표/스캔/표) (§5.2-④) |
| 블랙박스 유사도 정렬 | + **점수 분해·감사 + 조직 신호 랭킹** (§5.2-⑥) |
| 별도 검색 포털(탭 전환) | + **브라우저 사이드패널 상주 검색**(컨텍스트 전환 0) (§5.2-⑧) |
| 결과 올 때까지 화면 대기 | + **비동기 결과 + 포커스 없으면 OS(윈도우) 알림** (§5.2-⑨) |

### 5.2 차별화 9선

**① 크로스소스 엔티티·스레드 통합**
같은 인물·프로젝트·안건이 메일·게시물·결재에 서로 다른 식별자(메일주소·사번·한글/영문 이름)로 흩어져 있다. 별칭을 **표준 엔티티(canonical)** 로 정규화하고 공통 키로 소스를 가로질러 묶어, 개별 문서 나열이 아니라 **"이 안건의 메일 → 결재 → 공지" 한 묶음(맥락)** 으로 제시한다.
↳ *왜 다른가*: 대부분의 사내 검색은 문서를 독립적으로 반환한다. 무거운 GraphRAG 없이 **경량·결정론적 엔티티 해소**로 통합 맥락을 만든다.

**② 결정론적 메타·시간 추출 가드레일**
질의에서 기간("작년 4분기", "2024-03")·부서·문서종류·결재상태·작성자를 **코드(규칙)로 추출**해 *하드 필터*로 주입한다. LLM이 빠뜨리거나 잘못 채워도 방어된다.
↳ *왜 다른가*: 순수 임베딩 RAG는 "2024년 3분기"를 의미 벡터로만 처리해 날짜·숫자에서 부정확하다(업계가 말하는 *temporal hallucination*). 메타데이터를 정확도·보안의 control plane으로 다룬다.

**③ 환각 없는 "검색 우선" + 근거 부족 시 답변 보류(abstain)**
기본 모드는 출처 카드(생성 없음) → 빠르고 틀릴 수 없다. 요약 답변(생성)은 명시적으로 요청할 때만 수행하고, 그때도 **인용을 강제**하며 **유사도 임계 미달 시 "근거 부족"으로 답을 보류**한다.
↳ *왜 다른가*: 많은 RAG가 *항상* 생성하다 보니 자료에 없는 내용도 그럴듯하게 만들어내곤 한다. 업무 환경에서는 부담이 크기 때문에, '확실하지 않으면 단정하지 않는다'를 기본값으로 둔다.

**④ 멀티모달 첨부 의미 이해 (OCR 너머)**
스캔 결재·표·PPT 도표·이미지 첨부를 비전 모델(VLM)이 **한국어 설명문**으로 변환해 본문 임베딩에 합류시킨다 → "그래프가 보여주는 내용", "도장 찍힌 스캔본"까지 검색된다.
↳ *왜 다른가*: 대다수 시스템은 OCR로 뽑은 글자만 색인해 도표·레이아웃의 *의미*를 놓친다.

**⑤ 삭제·최신성 정합 (stale / 권한회수 방어)**
소스-인덱스 **공통 식별자** + 증분 동기화 + **삭제 전파**(원본에서 지워지면 인덱스에서도 즉시 제거, 권한 회수도 동기화)로 인덱스를 항상 원본과 일치시킨다.
↳ *왜 다른가*: 주기적 전체 재색인에만 기대면 삭제·권한 회수가 누락돼 **지워졌거나 권한이 없는 문서가 검색에 남을** 수 있다 — 업계에서도 특히 주의가 필요한 부분으로 꼽는 지점이다.

**⑥ 설명가능·감사가능 + 조직 신호 랭킹**
각 결과의 점수 분해(의미·키워드·조직 신호·최신성)를 노출해 "왜 이 순위인지" 설명하고, 모든 질의·열람을 감사 로그로 남긴다. 단순 유사도가 아니라 **최근성·열람 빈도·결재 완료·내 부서 관련성**으로 부스트해 "업무적으로 중요한" 순서를 만든다. 가중치는 정답셋(eval)으로 회귀 튜닝한다.
↳ *왜 다른가*: 블랙박스 검색 대비 신뢰·튜닝·컴플라이언스에 유리하고, 코사인 정렬만으로는 못 잡는 실무 우선순위를 반영한다.

**⑦ 한국어·한국 업무 특화 + 완전 온프레미스(데이터 주권)**
nori 형태소 + 한국어 리랭커 + **HWP/HWPX 파싱** + 사번·직급·부서·결재선 등 한국 조직 구조 이해. 로컬 LLM으로 질의·문서를 **외부로 내보내지 않는다**.
↳ *왜 다른가*: 글로벌 SaaS형 솔루션은 HWP·결재선·한글 조직도·데이터 주권 측면에서 상대적으로 약한 편이다. 규제·보안 환경에 자연스럽게 맞는다.

**⑧ 브라우저 사이드패널 상주 검색 (Side Panel API)**
크롬/엣지 확장의 **Side Panel API**(`chrome.sidePanel`, Manifest V3)로 검색 UI를 브라우저 우측 패널에 상주시킨다. 메일·그룹웨어·결재 화면을 **그대로 띄운 채** 옆 패널에서 검색·참조하므로 **탭 전환·별도 포털 이동이 0**이다. 패널은 현재 보고 있는 탭의 맥락(선택한 텍스트, 현재 결재 문서번호·게시글 URL 등)을 읽어 "지금 이 문서와 관련된 것" 검색·요약으로 바로 이어갈 수 있다(콘텐츠 스크립트 연동). 사내 브라우저 정책으로 확장을 **일괄 배포**하면 모든 직원이 어느 업무 화면에서든 즉시 사용 가능하다.
↳ *왜 다른가*: 대부분의 사내 검색은 "검색 포털 따로 접속" 방식이라 업무 흐름이 끊긴다. 사이드패널 상주 + 현재 화면 맥락 연동은 **업무 중단 없는(in-context) 검색**이라는 실사용 UX 차별점이다. (웹 앱 단독 접속·기존 시스템 임베드 위젯과도 병행 제공 가능.)
↳ *참고(범위/제약)*: 크로뮴 계열(Chrome 114+ · Edge) Side Panel API 의존 → 표준 채택 브라우저·확장 배포 정책 확인 필요(§14). 비지원 환경은 웹 앱으로 폴백.

**⑨ 비동기 결과 전달 — 브라우저 비활성 시 OS(윈도우) 알림**
무거운 질의(모드 B 요약 생성, 멀티소스 딥서치 등)는 즉시 끝나지 않을 수 있다. 사용자가 질의를 던지고 다른 일을 하면, 결과가 준비됐을 때 **브라우저·탭이 포커스가 아니어도 윈도우 OS 알림(toast)** 으로 알리고, 알림을 클릭하면 사이드패널/웹 앱이 결과 화면으로 포커스된다. 웹 앱은 Web Notifications API(`Notification`), 사이드패널 확장은 `chrome.notifications`(확장 서비스 워커)로 구현 — 후자는 브라우저가 최소화·비활성인 상태에서도 동작한다.
↳ *왜 다른가*: 대부분의 사내 검색은 "결과가 올 때까지 화면을 응시"하는 동기 방식이다. **fire-and-forget + OS 알림**은 온프레미스 LLM 생성·딥서치의 대기 시간을 체감상 가리고, 업무 중단 없는 사용 흐름(⑧)을 완성한다.
↳ *참고(범위/제약)*: OS 알림 권한 동의 필요, 사내 OS 알림 정책(집중 지원/방해 금지)에 영향받을 수 있음 → 인앱 배지·결과 큐로도 폴백. 짧은 질의(모드 A, 수초 이내)는 알림 없이 즉시 표시.

> 위 9가지는 §6 이후의 아키텍처·파이프라인·권한 설계로 구체화된다. ②③⑤는 §13 리스크의 핵심 실패 모드를 직접 방어하는 장치이고, ⑧⑨는 "업무 중단 없는" 접근·결과 전달 UX 로 §12 로드맵 후반(접근 채널 확장)과 맞물린다.

### 7.2 문서 처리

1. **포맷 파싱**: pdf(`pypdf`/PyMuPDF), docx(`python-docx`), xlsx(`openpyxl`), **hwp/hwpx** — 한국 특화 난점(§13 리스크). 이미지/스캔본 → **OCR**(한국어 지원 엔진).
2. **청킹**: 헤딩/문단 기준 의미 단위 분할 + **parent-child**(작은 청크로 검색 → 부모 섹션을 문맥으로 제공). 메타(작성자·일자·부서·문서종류·권한) 부착.
3. **임베딩**: 다국어 임베딩 모델 → 벡터 DB upsert (문서 단위 ID + 청크 단위 포인트).
4. **전문검색 색인**: 한국어 형태소(예: nori) BM25 인덱스에 본문/메타 색인.
5. **메타/권한 DB 적재**: 문서·청크·ACL·동기화 상태.

> 각 단계는 **증분·멱등**으로 설계 → 중단/재시작 안전. 대량 초기 인덱싱은 야간 배치.

## 9. 멀티유저 · 권한 · 보안 (가장 중요)

> 사내 시스템에서는 검색 품질 못지않게 **"권한이 새지 않는 것"**이 중요하다.

### 9.1 권한 (ACL) 모델

- 인덱싱 시 각 문서/청크에 **열람 허용 주체**(사용자ID/그룹/부서/결재선) 메타데이터 저장.
- 검색 시 로그인 사용자의 **소속 그룹·부서**를 권한 필터로 변환해 **벡터·키워드 검색 단계에서 pre-filter** (검색 결과 자체가 권한 내 문서만).
  - *post-filter(검색 후 거르기)는 위험*: top-k 가 권한 문서로 안 채워질 수 있고, 누락·누출 위험 → **pre-filter 원칙**.
- RAG 답변(모드 B)도 **권한 통과한 문서만** LLM 컨텍스트로 투입(인용도 권한 내 출처만).
- 권한 변경 동기화: 원본 시스템에서 권한이 바뀌면 인덱스 ACL 도 갱신(증분 동기화에 포함).

### 9.2 인증/세션

- SSO: 사내 **AD/LDAP / OIDC / SAML** 연동. 무인증 접근 금지.
- API 는 stateless(토큰 검증) → 수평 확장 용이.

### 9.3 보안/컴플라이언스

- **사내망 전용**, 외부 인터넷 차단.
- 전송 암호화(TLS), 저장 암호화(디스크/DB), 비밀정보(PII) 검색 로그 마스킹 검토.
- **감사 로그**: 누가·언제·무엇을 검색했고 어떤 문서를 열람했는지 기록(컴플라이언스·오남용 추적).
- **로컬 LLM 필수**: 질의·문서가 외부 API 로 나가지 않음.
- `[TBD]` 데이터 보존/파기 정책, 로그 보존 기간.

### 9.4 동시성

- API stateless + 다중 워커 → 수평 확장. LLM 동시성은 vLLM 배치로 처리량 확보.
- 임베딩/리랭커 서버 분리, 질의·임베딩 캐싱(권한 고려), rate limiting.

## 10. 구성 옵션 — 무GPU(검색형) vs GPU(생성형)

> GPU 는 본 시스템에서 비용이 가장 큰 요소다. 다행히 RAG 는 부품마다 GPU 필요도가 달라, **GPU 없이도 핵심 가치(통합 검색)를 먼저 제공**하고 이후 GPU 로 생성·멀티모달을 더하는 단계적 도입이 가능하다.

### 10.1 부품별 GPU 필요도

| 부품 | GPU 필요도 | 비고 |
|------|-----------|------|
| 벡터 검색(Qdrant/pgvector) | 불필요 | ANN 검색은 CPU 기반 |
| 키워드 검색(BM25/OpenSearch) | 불필요 | CPU |
| 질의 임베딩(검색어 1건) | CPU로 충분 | 단건이라 수십~수백 ms |
| 리랭커(cross-encoder) | CPU 가능(경량 ONNX) | 후보 수십 개 한정, 또는 생략 |
| 대량 초기 인덱싱 임베딩 | CPU 가능하나 느림 | 야간 배치로 흡수 |
| 답변 생성 LLM(모드 B) | 사실상 GPU 필요 | CPU 는 다중 사용자 실사용 곤란 |
| 멀티모달 VLM 캡션(§5.2-④) | GPU 권장 | CPU 는 매우 느림 |

즉 **검색해서 출처를 찾아 주는 RAG 의 핵심 절반은 GPU 없이도 동작**한다. GPU 가 꼭 필요한 부분은 *생성형 답변(모드 B)* 과 *멀티모달·대규모 동시성*이다.

### 10.2 두 가지 구성

| 구분 | 검색형 (무GPU) | 생성형 (GPU) |
|------|----------------|---------------|
| 제공 기능 | 통합 검색 + 출처 카드 + 코드 요약(건수·필터) | + 요약 답변(인용) + 멀티모달 캡션 |
| 대응 모드 | 모드 A | 모드 A + 모드 B |
| 하드웨어 | 기존 CPU 서버(GPU 0) | GPU 추가 (§11) |
| 환각 | 없음(생성 안 함) | 인용·근거 부족 시 보류로 관리 |
| 동시성 | 검색은 CPU 로 확장 용이 | LLM 은 vLLM 배치로 확보 |
| 적합 상황 | 빠른 도입·예산 최소·가치 검증 | 요약/정리 수요·시각자료 검색 |

### 10.3 권고 (단계적 도입 · 예산 de-risking)

- **1차는 검색형(무GPU)** 으로 출시해, 흩어진 문서를 한 번에 찾는 핵심 가치를 **GPU 0원**으로 먼저 입증한다.
- 효과가 확인되면 **생성형(GPU)** 으로 확장한다(요약 답변·멀티모달 캡션).
- 무GPU 의 유일한 약점인 **초기 대량 인덱싱 속도**는 ① 야간 배치 ② 경량 임베딩 모델 ③ 초기 1회만 GPU 단기 사용 후 증분은 CPU — 로 완화한다.

> 이 구분은 §12 로드맵(P1~P2 검색형 → P3 부터 생성형)과 §11 하드웨어 산정에 그대로 반영된다. 다만 **모드 B(생성)는 CPU 로는 다중 사용자 실사용이 어려워 사실상 GPU 가 필요**하다는 점은 분명히 해 둔다.

## 12. 단계별 로드맵 (마일스톤 초안)

각 단계는 *산출물 + 수락 기준*.

| 단계 | 내용 | 수락 기준(예) |
|------|------|----------------|
| **P0 환경** | Linux 서버, (검색형은 CPU만) LLM·임베딩·벡터DB·DB 기동, SSO 연동 PoC | 한국어 검색 1건 end-to-end (모드 A) |
| **P1 단일 소스 MVP** | 가장 쉬운 소스 1종(예: 게시물) 커넥터 + 인덱싱 + 하이브리드 검색 + 권한 pre-filter — **검색형(무GPU)** | 권한 지킨 검색, 정답률·지연 기준 통과 |
| **P2 멀티소스** | 메일 + 결재문서 커넥터, 첨부 파싱/OCR(hwp 포함), 증분 동기화 — 여전히 **무GPU 가능** | 3종 통합 검색, 증분 동기 안정 |
| **P3 RAG 답변** | 리랭커 + 모드 B(요약+인용), 환각/권한 누출 가드 — **GPU 도입 지점** | 인용 정확, 권한 누출 0 |
| **P4 멀티유저/운영** | 동시성·부하 테스트, 모니터링, 감사 로그, 백업/복구 | SLA 충족, 부하 테스트 통과 |
| **P5 확장(선택)** | **브라우저 사이드패널 확장(§5.2-⑧) + 비동기 OS 알림(§5.2-⑨)**, 멀티모달 캡션, 추가 소스, K8s 오토스케일 | 사이드패널 맥락 검색 + 비활성 시 알림 동작 |

> **GPU 관점**: P1~P2 검색형은 **무GPU**(기존 CPU 서버)로 진행할 수 있고, GPU 는 **P3 생성형부터** 필요하다(§10). 접근 채널도 **P1 웹 앱(단독 접속)** 으로 시작해, 검색이 안정된 뒤 **P5 에서 사이드패널**로 확장한다 — 사이드패널은 별도 백엔드 없이 동일 API 를 호출하는 가벼운 확장이다.

## 14. 결정 필요 사항 (검토 회의 입력)

> 보강 단계에서 이 값들을 채우면 스펙·일정·비용이 구체화된다.

1. **규모**: 총 사용자 수 / 피크 동시 접속 / 총 문서·첨부 수 및 증가율.
2. **데이터 소스 제품**: 메일 솔루션, 그룹웨어/게시판 제품, 전자결재 솔루션(자체개발 여부, API 제공 여부).
3. **권한 체계**: AD/LDAP 사용 여부, 부서/결재선 구조, 문서별 ACL 추출 가능 여부.
4. **검색 모드**: 모드 A 만 / A→B 단계 / 처음부터 B 포함.
5. **인프라 표준**: 온프레미스 vs 사내 클라우드, Kubernetes 사용 여부, 표준 DB·스토리지·모니터링·백엔드/프론트 프레임워크.
6. **GPU 예산/조달**: 가능한 GPU 등급 → 채택 가능 모델 크기 결정.
7. **SLA**: 검색/RAG 응답 목표, 가용성(이중화) 요구.
8. **보안/컴플라이언스**: PII 처리, 로그·데이터 보존 기간, 감사 요건.
9. **저장소 형태**: 통합형(PostgreSQL+pgvector) vs 전용형(Qdrant + OpenSearch).
10. **접근 채널**: 사내 표준 브라우저(크롬/엣지 Side Panel API 지원 여부·버전), 확장 일괄 배포 정책(MDM/그룹정책), 웹 앱 단독 제공 범위.
