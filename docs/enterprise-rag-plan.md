## 0. 한 줄 요약

> 사내에 흩어진 **메일 · 게시물 · 결재문서**를, 권한을 지키면서 **자연어 한 문장으로 통합 검색**하는 사내 전용 RAG 시스템.
> 사내망·온프레미스 전용으로, 데이터를 외부로 내보내지 않는다.

### 용어 빠른 정의 (AI/ML 입문자용)

- **RAG (Retrieval-Augmented Generation)**: "먼저 검색해서 찾은 문서를 LLM 에게 보여주고 답하게 한다." LLM 단독이 아니라 *DB 검색 + LLM 종합*.
- **임베딩 (embedding)**: 문장을 의미를 담은 숫자 벡터(예: 1024차원)로 바꾼 것. 의미가 비슷하면 벡터도 가깝다 → "예산 회의" 와 "비용 검토" 를 키워드 없이도 연결.
- **하이브리드 검색**: 키워드 검색(BM25)과 의미 검색(벡터)을 **둘 다** 돌려 합치는 것. 정확한 이름·번호는 키워드가, 자연어 의도는 벡터가 잡는다.
- **리랭커 (reranker)**: 1차로 넓게 찾은 후보 수십 개를, 질의와 각 문서를 **함께** 정밀 비교해 다시 순위 매기는 모델(cross-encoder). 검색 정확도를 크게 올리는 RAG 표준 단계.
- **청킹 (chunking)**: 긴 문서를 검색 단위(문단/섹션)로 쪼개는 것. 결재문서·긴 메일은 통째로 임베딩하면 정확도가 떨어져 쪼갠다.

## 1. 배경 & 문제

- 사내 업무 정보가 **시스템별로 단절**되어 있다: 메일(메일서버), 공지·게시물(그룹웨어/게시판), 결재문서(전자결재 시스템).
- 찾을 때마다 시스템을 옮겨 다니며 키워드로 검색 → 정확한 제목/작성자/기간을 기억해야 하고, 자연어 의도("작년 4분기 예산 관련 결재 건")로는 못 찾는다.
- 결과적으로 **이미 사내에 있는 정보를 다시 묻거나, 못 찾아서 중복 작업**이 발생한다.
- 사내 데이터는 **외부 SaaS LLM 에 올릴 수 없다**(보안·컴플라이언스) → **온프레미스 로컬 LLM** 필수.

## 2. 목표 & 범위

### 2.1 목표

1. 메일·게시물·결재문서를 **한 검색창**에서 자연어로 검색.
2. 결과에 **출처(원문 링크·작성자·일자·시스템)**를 항상 표기, 가능하면 **요약 답변 + 인용**.
3. **권한을 100% 준수** — 사용자가 접근 권한 없는 문서는 검색·답변·요약 어디에도 노출 금지.
4. **멀티유저 동시 접속**에서 만족스러운 응답 속도(목표 SLA 는 §8).
5. 사내망 전용, 외부 인터넷 노출 금지(전송·저장 암호화, 감사 로그).

### 2.2 범위 (In Scope)

- 데이터 소스: **메일 · 게시물 · 결재문서** (1차). 첨부파일(pdf/docx/xlsx/hwp/이미지) 텍스트 추출 포함.
- 검색 모드: ① 검색(문서 카드+스니펫) ② RAG 답변(요약+인용) — 둘 다 또는 단계 도입(§4.2).
- 권한 기반 필터링, SSO 로그인, 감사 로그.

### 2.3 비범위 (Out of Scope, 1차 제외 — 결정 필요)

- 실시간 채팅/그룹웨어 대체, 문서 작성·결재 기능(검색 전용).
- 외부(인터넷) 정보 검색.
- `[TBD]` 음성 검색, 모바일 전용 앱, 다국어 UI 등.

## 3. 사용자 시나리오 (예시)

| 질의 예 | 기대 동작 |
|---------|-----------|
| "지난 분기 마케팅 예산 결재 문서 찾아줘" | 결재문서, 기간(분기)·부서 필터 + 의미검색 → 카드 + 요약 |
| "김부장이 보낸 보안 점검 관련 메일" | 메일, 발신자=김부장 + 주제 의미검색 |
| "재택근무 정책 최신 공지" | 게시물, 의미검색 + 최신순 |
| "이 프로젝트 관련 메일·결재·게시물 다 모아줘" | 소스 통합 검색, 프로젝트명 의미검색 |
| "작년 12월 출장비 정산 어떻게 했더라" | 결재+메일, 기간 필터 + RAG 요약(인용) |

> **권한 전제**: 위 모든 질의는 *질의한 사용자가 볼 수 있는 문서 안에서만* 동작한다.

## 4. 핵심 설계 원칙

### 4.1 검색 코어

- **하이브리드 검색**: 의미 검색(벡터)과 키워드 검색(BM25, 한국어 형태소)을 함께 수행하고 **RRF(Reciprocal Rank Fusion)** 로 결합. 정확한 이름·번호는 키워드가, 자연어 의도는 벡터가 보완한다.
- **리랭커 추가**: RRF 상위 후보를 cross-encoder 로 재정렬해 정확도를 끌어올린다.
- **권한 우선(permission-first)**: 검색의 모든 단계에서 사용자가 볼 수 있는 문서만 후보로 둔다(§7).
- **증분·멱등 인덱싱**: 변경분만 반영하고 단계별 재실행이 안전하도록 설계 → 대량 초기 인덱싱과 운영 동기화를 안정적으로.
- **로컬 LLM**: 질의·문서가 외부로 나가지 않도록 온프레미스 추론.

> 본 시스템에서 새로 풀어야 하는 핵심 과제는 **① 다양한 사내 소스 연동 ② 권한 정확성 ③ 멀티유저 성능** 세 가지다. 검색 알고리즘 자체보다 이 셋이 난이도를 좌우한다.

### 4.2 검색 모드 (2가지 — 단계 도입 가능)

- **모드 A: 검색(Retrieval-only)** — 문서 카드 + 강조 스니펫 + 출처. LLM 답변 없음(또는 1줄 요약). 빠르고 안전(환각 없음).
- **모드 B: RAG 답변(Generation)** — 상위 문서를 LLM 에 넣어 **요약 답변 + 인용 출처**. "요약해줘/정리해줘" 류에 유용하나 **환각·권한 누출 위험**이 있어 인용 강제·권한 필터 후처리가 필수.
- **결정 필요**: 1차는 모드 A 로 안전하게 출시 → 검증 후 모드 B 추가(권장). 또는 처음부터 B 포함.

## 6. 데이터 파이프라인 (소스 → 검색 가능 인덱스)

### 6.1 소스 커넥터 (증분 동기화)

| 소스 | 연동 방식 후보 | 추출 대상 | 비고 |
|------|----------------|-----------|------|
| 메일 | Exchange(EWS/Graph) · IMAP · 메일 아카이브 DB | 제목/본문/발신·수신/일자/첨부 | `[TBD]` 사내 메일 솔루션 종류 |
| 게시물 | 게시판/그룹웨어 DB 직접 · REST API | 제목/본문/작성자/게시판/일자/첨부 | `[TBD]` 그룹웨어 제품 |
| 결재문서 | 전자결재 시스템 API/DB | 문서/기안자/결재선/부서/일자/본문/첨부 | `[TBD]` 전자결재 솔루션(자체/상용) |

- **증분 전략**: 변경분만 수집 — 타임스탬프 폴링 / CDC / webhook(시스템 지원 시). 마지막 처리 커서를 저장해 재실행 시 이어받기.
- **권한 메타데이터 동시 수집**: 각 문서의 **열람 허용 주체(사용자/그룹/부서/결재선)**를 함께 가져와 인덱스에 저장 → §7 권한 필터의 근거.

### 6.2 문서 처리

1. **포맷 파싱**: pdf(`pypdf`/PyMuPDF), docx(`python-docx`), xlsx(`openpyxl`), **hwp/hwpx** — 한국 특화 난점(§10 리스크). 이미지/스캔본 → **OCR**(한국어 지원 엔진).
2. **청킹**: 헤딩/문단 기준 의미 단위 분할 + **parent-child**(작은 청크로 검색 → 부모 섹션을 문맥으로 제공). 메타(작성자·일자·부서·문서종류·권한) 부착.
3. **임베딩**: 다국어 임베딩 모델 → 벡터 DB upsert (문서 단위 ID + 청크 단위 포인트).
4. **전문검색 색인**: 한국어 형태소(예: nori) BM25 인덱스에 본문/메타 색인.
5. **메타/권한 DB 적재**: 문서·청크·ACL·동기화 상태.

> 각 단계는 **증분·멱등**으로 설계 → 중단/재시작 안전. 대량 초기 인덱싱은 야간 배치.

### 7.2 "단순화 옵션" (소규모/빠른 PoC)

운영 부담을 줄이려면 저장소를 **PostgreSQL 단일**로 통합 가능:
- 메타/권한/감사 = PostgreSQL, 벡터 = **pgvector**, 전문검색 = PostgreSQL 한국어 FTS.
- 장점: 운영 컴포넌트 최소(엔진 1개). 단점: 초대규모·고난도 한국어 검색 품질은 OpenSearch(nori)·전용 벡터DB 대비 열위.
- **결정 필요**: 규모/품질 요구에 따라 [통합형(pgvector)] vs [전용형(Qdrant+OpenSearch)] 선택.

## 8. 멀티유저 · 권한 · 보안 (가장 중요)

> 사내 시스템의 성패는 검색 품질만큼이나 **"권한을 절대 새지 않는가"**에 달렸다.

### 8.1 권한 (ACL) 모델

- 인덱싱 시 각 문서/청크에 **열람 허용 주체**(사용자ID/그룹/부서/결재선) 메타데이터 저장.
- 검색 시 로그인 사용자의 **소속 그룹·부서**를 권한 필터로 변환해 **벡터·키워드 검색 단계에서 pre-filter** (검색 결과 자체가 권한 내 문서만).
  - *post-filter(검색 후 거르기)는 위험*: top-k 가 권한 문서로 안 채워질 수 있고, 누락·누출 위험 → **pre-filter 원칙**.
- RAG 답변(모드 B)도 **권한 통과한 문서만** LLM 컨텍스트로 투입(인용도 권한 내 출처만).
- 권한 변경 동기화: 원본 시스템에서 권한이 바뀌면 인덱스 ACL 도 갱신(증분 동기화에 포함).

### 8.2 인증/세션

- SSO: 사내 **AD/LDAP / OIDC / SAML** 연동. 무인증 접근 금지.
- API 는 stateless(토큰 검증) → 수평 확장 용이.

### 8.3 보안/컴플라이언스

- **사내망 전용**, 외부 인터넷 차단.
- 전송 암호화(TLS), 저장 암호화(디스크/DB), 비밀정보(PII) 검색 로그 마스킹 검토.
- **감사 로그**: 누가·언제·무엇을 검색했고 어떤 문서를 열람했는지 기록(컴플라이언스·오남용 추적).
- **로컬 LLM 필수**: 질의·문서가 외부 API 로 나가지 않음.
- `[TBD]` 데이터 보존/파기 정책, 로그 보존 기간.

### 8.4 동시성

- API stateless + 다중 워커 → 수평 확장. LLM 동시성은 vLLM 배치로 처리량 확보.
- 임베딩/리랭커 서버 분리, 질의·임베딩 캐싱(권한 고려), rate limiting.

### 9.2 규모별 하드웨어 (초안 — vLLM + 양자화(AWQ/GPTQ) 가정)

| 규모 | 전체 사용자 / 피크 동시 | LLM/임베딩 GPU | API·DB·검색 노드 | 스토리지 |
|------|--------------------------|----------------|-------------------|----------|
| **PoC/소규모** | ~50 / 5~10 | GPU 1× 24GB (예: RTX 4090 / L4 / A5000), 7~8B Q4 + 임베딩 | 16 vCPU / 64GB RAM (단일 서버 동거 가능) | NVMe 1~2TB |
| **중규모** | ~500 / 30~50 | GPU 1~2× 40~48GB (A100 40GB / L40S 48GB), 14B + 임베딩/리랭커 분리 | 32 vCPU / 128~256GB, 검색·DB 별도 노드 | NVMe 4TB+ |
| **대규모** | ~5,000 / 100+ | GPU 다수(A100/H100), LLM·임베딩 풀 분리 + 오토스케일 | API/검색/DB 분리 + K8s + 복제 | 수십 TB + 백업 |

핵심 고려:
- **GPU VRAM 이 모델 크기와 동시 처리량을 좌우한다**: 7~8B 모델 + 임베딩이 24GB 급에 안정적으로 들어가는 수준 → 더 큰 모델이나 높은 동시성이 필요하면 40GB 이상 권장. vLLM 은 VRAM 여유가 클수록 동시 처리량↑.
- **임베딩/리랭커는 별 GPU/배치 서버**로 분리하면 LLM 지연과 간섭 최소화.
- **벡터 DB 메모리**: 문서·청크 수 × 차원 × 4byte + 인덱스. 데이터량 추정 후 RAM 산정(`[TBD: 총 문서/청크 수]`).
- **CPU/RAM**: 파싱·OCR·동기화 배치는 CPU 부하 큼 → 인덱싱 노드 분리 고려.
- 온프레미스 vs 사내 클라우드(GPU 인스턴스) `[TBD]`.

> **결정 필요(스펙 산정 입력값)**: ① 총 사용자/피크 동시 ② 총 문서·첨부 수와 증가율 ③ 응답 SLA ④ GPU 예산/조달 가능 모델 ⑤ 기존 인프라(쿠버네티스·NAS·DB 표준).

## 12. 결정 필요 사항 (검토 회의 입력)

> 보강 단계에서 이 값들을 채우면 스펙·일정·비용이 구체화된다.

1. **규모**: 총 사용자 수 / 피크 동시 접속 / 총 문서·첨부 수 및 증가율.
2. **데이터 소스 제품**: 메일 솔루션, 그룹웨어/게시판 제품, 전자결재 솔루션(자체개발 여부, API 제공 여부).
3. **권한 체계**: AD/LDAP 사용 여부, 부서/결재선 구조, 문서별 ACL 추출 가능 여부.
4. **검색 모드**: 모드 A 만 / A→B 단계 / 처음부터 B 포함.
5. **인프라 표준**: 온프레미스 vs 사내 클라우드, Kubernetes 사용 여부, 표준 DB·스토리지·모니터링·백엔드/프론트 프레임워크.
6. **GPU 예산/조달**: 가능한 GPU 등급 → 채택 가능 모델 크기 결정.
7. **SLA**: 검색/RAG 응답 목표, 가용성(이중화) 요구.
8. **보안/컴플라이언스**: PII 처리, 로그·데이터 보존 기간, 감사 요건.
9. **저장소 형태**: 통합형(PostgreSQL+pgvector) vs 전용형(Qdrant + OpenSearch).
